# 反向预取

先回顾关键背景：分组结构
根据文档前面的设定：
fully_shard(m2) → m2 独立成组，参数 b
fully_shard(m3) → m3 独立成组，参数 c
fully_shard(model) → 根组（Root group），包含剩余的所有参数，即 m1 的 a 和 m4 的 d
重点：m4 和 m1 属于同一个组（根组），它们的参数 (a,d) 是作为一个整体被 all-gather 和 reduce-scatter 的。

反向传播的过程

```text
              time ──────────────────────────────────────────────►

compute:      [ bwd(m4,m3)     | bwd(m2)        | bwd(m1)       ]
AG stream:    [AG(c)] [ AG(b)  |   AG(a,d)      ]
RS stream:                     |[RS(c)]  [ RS(b)|     RS(a,d)   ]
```

## 问题一：为什么 m4 的反向不需要 all-gather？

前向传播的最后阶段

```text
前向顺序: m1 → m2 → m3 → m4

1. Root pre-forward:  AG(a,d)   ← 为 m1 和 m4 all-gather 参数
2. fwd(m1)                       ← 使用 a
3. m2 pre-forward:    AG(b)
4. fwd(m2)                       ← 使用 b
5. m3 pre-forward:    AG(c)
6. fwd(m3)                       ← 使用 c
7. fwd(m4)                       ← 使用 d（d 属于根组，在第1步已 all-gather）
8. Root post-forward: reshard(a,d) ← 释放 (a,d) 的完整参数
```

反向传播的起始阶段
前向结束后立刻进入反向。当 m4 的 pre-backward hook 触发时，它要求根组 unshard (a,d)。但这里有一个关键的内存管理细节：
FSDP2 的 post_forward_reshard 并不会真正释放 GPU 显存。它只是将完整的参数缓冲标记为"可重用"（通过 FSDP2 自己的内存管理器，而非 torch.Tensor.record_stream）。当 m4 的 pre_backward_unshard 立刻紧随其后触发时：
缓冲仍在 GPU 显存中，尚未被其他操作覆盖
FSDP2 的内存管理器重用同一块缓冲
all-gather 可以直接"原地"写入（数据实际上已经是完整的了），通信量极小或本质上是一个空操作（no-op）
所以在图中，m4 的反向计算前面没有显式的 AG 操作——它的参数在前向最后阶段就已经是完整状态，且在进入反向时缓冲仍然可用。
简单来说：前向刚用完 → 反向马上要用 → 缓冲还在 → 不需要重新 all-gather。

## 问题二：为什么写成 bwd(m4,m3)？

把 m4 和 m3 合并在一起写，并不是说它们属于同一个组，而是说明 它们在 GPU 计算流上是紧密连续执行的，中间几乎没有空隙。具体原因如下：

### 反向传播的完整流程（GPU 视角）

```text
              time ──────────────────────────────────────────────►

compute:      [ bwd(m4,m3)     | bwd(m2)        | bwd(m1)       ]
AG stream:    [AG(c)] [ AG(b)  |   AG(a,d)      ]
RS stream:                     |[RS(c)]  [ RS(b)|     RS(a,d)   ]
```

详细拆解每一步：

| 时间 | 计算流 (compute) | AG 流 | RS 流 | CPU 侧 |
| --- | --- | --- | --- | --- |
| T1 | bwd(m4) | AG(c)（为 m3 预取） | — | CPU 触发 m4 的 pre-bwd hook，m4 的 unshard 是空操作，同时触发 m3 的 AG(c) |
| T2 | bwd(m3)（紧随 m4 之后） | AG(b)（为 m2 预取） | — | m4 的 post-bwd 暂不执行 RS（见下文），CPU 触发 m3 的 pre-bwd，m3 的 AG 已完成，同时预取 m2 的 AG(b) |
| T3 | bwd(m2) | AG(a,d)（为 m1 预取） | RS(c) | m3 的 post-bwd 触发 RS(c)，同时预取 m1 的 AG(a,d) |
| T4 | bwd(m1) | — | RS(b) | m2 的 post-bwd 触发 RS(b) |
| T5 | — | — | RS(a,d) | m1 的 post-bwd 触发 RS(a,d) |

m4 的 unshard 是空操作：如上面解释的，(a,d) 的缓冲还在，不需要等待 all-gather
m3 的 AG(c) 可以立刻发起：在 bwd(m4) 还在 GPU 上运行时，CPU 已经提前触发了 m3 的 pre-backward hook，在 AG 流上发起 AG(c)
当 bwd(m4) 完成时，AG(c) 已经完成了，所以 bwd(m3) 可以无缝衔接，不需要等待
在 GPU 视角上，m4 和 m3 的反向计算几乎是背靠背执行的，所以画成一个连续的块

## 对比：为什么 bwd(m2) 和 bwd(m1) 是分开的？

因为 bwd(m1) 属于根组，它需要等 AG(a,d) 完成。虽然 AG(a,d) 在 bwd(m2) 期间就已经在 AG 流上预取了，但从图的布局来看，每个组之间的转换都有一个明确的通信边界，所以分开画更清晰。
而 m4 和 m3 之间没有这个通信边界（m4 不需要通信，m3 的通信被完全隐藏了），所以合并画。

## 问题三：为什么RS流是这样的？

```text
              time ──────────────────────────────────────────────►

compute:      [ bwd(m4,m3)     | bwd(m2)        | bwd(m1)       ]
AG stream:    [AG(c)] [ AG(b)  |   AG(a,d)      ]
RS stream:                     |[RS(c)]  [ RS(b)|     RS(a,d)   ]
```

为什么 RS(c) 紧跟在 bwd(m3) 后面，RS(b) 紧跟在 bwd(m2) 后面，而 RS(a,d) 却要等到最后 bwd(m1) 跑完之后才执行？为什么 m4 跑完反向后不立刻 RS(a,d)？
这涉及 FSDP 中关于 “组（Group/Unit）” 和 “钩子（Hook）触发时机” 的核心机制。

1 FSDP 的 Reduce-Scatter 触发规则

- 在 FSDP 中，反向传播时的通信是通过 hook 来触发的：
Pre-backward hook：在模块反向计算前触发，负责 all-gather（收集完整参数以计算梯度）。
Post-backward hook：在模块反向计算后触发，负责 reduce-scatter（将计算出的梯度分片并释放完整参数）。
最核心的规则是：reduce-scatter 是针对“组（Group）”级别的，而不是“单个模块”级别的。
一个组的 reduce-scatter 必须等到该组内所有模块的反向计算都完成后，才能统一执行。

2.反向传播的时间线拆解

- 反向传播的顺序是：m4 -> m3 -> m2 -> m1。

| 执行步骤 | 计算流 (Compute) | 发生了什么？ (Hook 触发与通信) | 为什么？ |
| :--- | :--- | :--- | :--- |
| Step 1 | bwd(m4) | m4 计算出了 d 的梯度。但 m4 属于根组 (a,d)，此时 m1 还没算，根组的梯度没凑齐，不能执行 RS(a,d)。 | 必须等同组内的 m1 也算完。 |
| Step 2 | bwd(m3) | m3 计算出了 c 的梯度。m3 是独立组，它的梯度算完了，立刻触发 `RS(c)`。 | 独立组，算完就能立刻通信。 |
| Step 3 | bwd(m2) | m2 计算出了 b 的梯度。m2 是独立组，立刻触发 `RS(b)`。 | 独立组，算完就能立刻通信。 |
| Step 4 | bwd(m1) | m1 计算出了 a 的梯度。此时，根组 (a,d) 中的 m4 和 m1 都算完了！ 根组的梯度终于凑齐，触发 `RS(a,d)`。 | 根组内所有模块反向完毕，可以统一通信了。 |

3.对照原图理解

```text
              time ──────────────────────────────────────────────►

compute:      [ bwd(m4,m3)     | bwd(m2)        | bwd(m1)       ]
AG stream:    [AG(c)] [ AG(b)  |   AG(a,d)      ]
RS stream:                     |[RS(c)]  [ RS(b)|     RS(a,d)   ]
```

- RS(c) 和 RS(b)：因为 m3 和 m2 都是独立包装（fully_shard）的子模块，它们各自形成一个独立的通信组。一旦它们的反向计算完成，就可以立刻在 RS 流上发起自己的 reduce-scatter，从而与后续的计算重叠。
RS(a,d) 被推迟到最后：因为 m4 和 m1 没有被单独包装，它们“漏”到了根组里。FSDP 必须等反向传播一路算到 m1，把 m1 的梯度也算出来后，才能把 m4 和 m1 的梯度打包在一起，执行一次大的 RS(a,d)。

4.这揭示了什么优化建议？

- 文档中提到的 "This pipelining is why the recommended pattern is to apply fully_shard bottom-up to each layer"（正是由于这种流水线机制，推荐自底向上对每一层应用 fully_shard），正是为了解决你发现的这个问题！
如果你只对 m2 和 m3 做了 fully_shard，而把 m1 和 m4 留给了根组，就会导致：
m4 的梯度在显存里“干等”：m4 算完梯度后，因为要等 m1，它的完整梯度张量必须一直留在显存里，直到最后 RS(a,d) 执行完才能释放，这浪费了显存。
通信无法重叠：最后的 RS(a,d) 是一个包含 a 和 d 的大通信操作，且发生在所有计算结束之后，无法与计算重叠。
正确的做法（自底向上包装每一层）：
如果你把 m1 和 m4 也分别包装：fully_shard(m1), fully_shard(m4)，那么它们就会变成独立组。
m4 反向一结束，立刻执行 RS(d)，释放 d 的完整梯度显存。
m1 反向一结束，立刻执行 RS(a)。
这样所有的 reduce-scatter 都能像 RS(c) 和 RS(b) 一样，完美地与计算流重叠，且显存占用最小！
