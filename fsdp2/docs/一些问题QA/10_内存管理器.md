# FSDP2 内存管理器

## 问题一：FSDP2 的内存管理器是什么？它解决了什么问题？

FSDP2 实现了一套**自己的内存管理机制**，核心目标是在多个 CUDA Stream 并行执行的环境中，**安全地分配和释放 all-gather 输出缓冲区**，同时保证：

1. **正确性**：一个 Stream 正在使用的内存，不会在另一个 Stream 还持有时被回收/覆写
2. **确定性**：内存使用模式可预测，不会因为 CUDA 缓存分配器（Caching Allocator）的内部状态变化而产生不确定行为
3. **性能**：最大化计算与通信的重叠（overlap），不引入不必要的 CPU 阻塞

### 背景：FSDP2 的多 Stream 并行架构

FSDP2 将通信与计算分离到不同的 CUDA Stream 上并发执行：

```text
                       time ──────────────────────────────────────►

Compute Stream:  [ fwd(m1)         | fwd(m2)         | fwd(m3)    ]
AG Stream:       [ AG(a,d)  | AG(b)         | AG(c)               ]
Copy-in Stream:  [ copy-in  | copy-in       | copy-in             ]
RS Stream:       [          |               | RS(c) | RS(b) | ... ]
```

- **Compute Stream**（默认流）：前向/反向计算
- **All-Gather Stream**：执行 NCCL all-gather 集合通信
- **All-Gather Copy-In Stream**：将分片参数拷贝到 all-gather 输入缓冲区
- **Reduce-Scatter Stream**：执行梯度 reduce-scatter
- **All-Reduce Stream**（HSDP）：层级式 all-reduce

这些 Stream 并发执行，意味着**一块内存可能同时被多个 Stream 引用**。例如：all-gather 的输出缓冲区在 AG Stream 上被写入后，Compute Stream 上正在读它做前向计算，同时 Copy-In Stream 可能正准备把它回收给下一层复用。

### FSDP2 内存管理器的核心思路

```text
┌──────────────────────────────────────────────────────────┐
│                   FSDP2 Memory Manager                     │
│                                                          │
│  ┌─────────────────┐    ┌──────────────────┐             │
│  │ Storage Resizing │    │ CUDA Event Sync  │             │
│  │ (alloc/free)     │    │ (record & wait)  │             │
│  └────────┬────────┘    └────────┬─────────┘             │
│           │                      │                        │
│           ▼                      ▼                        │
│  ┌───────────────────────────────────────────┐           │
│  │     跨 Stream 安全的生命周期管理            │           │
│  │  • 分配：all_gather_output 按需 resize     │           │
│  │  • 释放：storage.resize_(0) 保留 tensor    │           │
│  │  • 同步：Event 保证 happens-before 关系    │           │
│  └───────────────────────────────────────────┘           │
└──────────────────────────────────────────────────────────┘
```

**不需要 `record_stream`**。FSDP2 完全通过 CUDA Event 手动控制同步点，确保内存的分配/释放在正确的时机发生。

## 问题二：FSDP2 如何管理内存的分配和释放？

### 2.1 Storage Resizing：不创建新 Tensor，只缩放 Storage

FSDP2 使用 **storage resizing** 来管理内存，而不是每次都创建新的 Tensor。核心原理：

```python
# 文件：torch/distributed/fsdp/_fully_shard/_fsdp_param.py
# FSDPParam 中的关键方法

def alloc_storage(self, tensor: torch.Tensor) -> None:
    """分配存储空间：只在需要时 resize"""
    storage = tensor.untyped_storage()
    size = tensor.numel() * tensor.element_size()
    if storage.size() != size:
        storage.resize_(size)

def free_storage(self, tensor: torch.Tensor) -> None:
    """释放存储空间：resize 到 0"""
    storage = tensor.untyped_storage()
    storage.resize_(0)
```

**为什么要用 storage resizing 而不是创建新 Tensor？**

```text
传统做法（❌）：
  每次 all-gather 都 new_tensor → PyTorch allocator 分配新内存
  旧 tensor 被 GC → allocator 异步回收
  问题：autograd 可能持有旧 tensor 的引用，释放可能导致 use-after-free

FSDP2 做法（✓）：
  Tensor 对象在初始化时创建一次，之后只 resize 其底层 storage
  alloc:  storage.resize_(N) → 分配 N 字节
  free:   storage.resize_(0) → 释放所有字节（tensor 对象仍在）
  好处：autograd 持有的引用仍然有效（同一个 tensor 对象），只是大小变了
```

### 2.2 All-Gather 输出的完整生命周期

以一个参数组为例，观察 all-gather 输出缓冲区从创建到释放的完整过程：

```text
时间轴 ──────────────────────────────────────────────────────────►

阶段1: 初始化 (init_all_gather_outputs)
  │
  ├─ 为每个参数创建一个 1D 空 tensor，大小为 numel * world_size
  │  storage 初始大小为 0（不占显存）
  │
  ▼
阶段2: 分配 (alloc_all_gather_outputs) — 在 all-gather 之前
  │
  ├─ 对每个 output tensor 调用 alloc_storage()
  │  storage.resize_(numel * world_size * element_size)
  │  此时才真正占用 GPU 显存
  │
  ▼
阶段3: 使用 (all-gather copy-in + comm + copy-out)
  │
  ├─ Copy-In Stream:  将分片参数 copy 到 output 的对应 rank 位置
  ├─ AG Stream:      NCCL all-gather 填满整个 output
  ├─ Compute Stream: 通过 as_strided 创建 unsharded view，用于计算
  │
  ▼
阶段4: 释放 (free_unsharded_param) — 在 reshard 时
  │
  ├─ 对所有 output tensor 和 inner tensor 调用 free_storage()
  │  storage.resize_(0)
  │  显存被归还给 PyTorch caching allocator
  │
  ▼
阶段5: 下次迭代 → 回到阶段2（复用同一个 tensor 对象）
```

### 2.3 代码视角：unshard 和 reshard

```python
# 简化后的 unshard 流程 (FSDPParamGroup.pre_forward → unshard → wait_for_unshard)

def unshard(self):
    # 1. 分配 all-gather 输出缓冲区
    for param in self.fsdp_params:
        param.init_all_gather_outputs()     # tensor 对象已存在
        param.alloc_all_gather_outputs()    # storage.resize_() 分配内存
    
    # 2. 发起 all-gather 通信
    # Copy-In Stream: copy sharded data → output buffer
    # AG Stream: NCCL all-gather
    all_gather_result = foreach_all_gather(...)
    return all_gather_result

def wait_for_unshard(self):
    # 3. 等待 all-gather 完成
    # Compute Stream wait on AG Stream event
    
    # 4. 创建 unsharded parameter view
    for param in self.fsdp_params:
        param.init_unsharded_param()  # as_strided 从 output buffer 创建 view

# 简化后的 reshard 流程 (FSDPParamGroup.reshard → post_forward/post_backward)

def reshard(self):
    for param in self.fsdp_params:
        param.to_sharded()  # 1. 将 sharded param 注册回 module
                            # 2. 调用 free_unsharded_param()
                            #    → output storage.resize_(0)
                            #    → inner tensor storage.resize_(0)
```

### 2.4 关键优化：跨层复用缓冲区

考虑一个实际的前向传播序列。FSDP2 的 `AllGatherState` 机制让相邻层之间可以**无缝接力**使用 all-gather 结果：

```text
时间轴 ──────────────────────────────────────────────────────────►

Layer 1:  [alloc] [AG(m1)] [copy-out] [fwd(m1) .................] [flush]
                                                │
Layer 2:                        [alloc] [AG(m2)] [copy-out] [fwd(m2)...]
                                        ▲
                                        │
                        上一层的 AllGatherState.event
                        保证: Layer2 的 alloc 不会在 Layer1 还在用时回收内存
```

**FSDPCommContext 中的 `all_gather_state`**：

```python
# 文件：torch/distributed/fsdp/_fully_shard/_fsdp_comm.py (概念代码)

AllGatherState = namedtuple("AllGatherState", [
    "all_gather_result",   # 上一层的 all-gather 输出
    "event",               # all-gather copy-out 完成的事件
])

# 在 param_group.wait_for_unshard() 中:
prev_state = self.comm_ctx.all_gather_state
if prev_state is not None:
    # 等上一层的 all-gather copy-out 完全结束后，才释放它的输出
    self._wait_all_gather_streams_on_event(prev_state.event)
    self.comm_ctx.all_gather_state = None  # 释放上一层的引用
```

这保证了：上一层（如 Layer 1）的 all-gather 输出缓冲区在它还挂着 `AllGatherState` 引用时，**不会被 PyTorch caching allocator 回收给其他用途**。只有当下一层（Layer 2）的 copy-in 完成后，上一层的引用才被清除，此时如果 Python 侧没有其他引用，内存才会被回收。

## 问题三：为什么 FSDP2 彻底放弃 `record_stream`？

### 3.1 FSDP1 的 `record_stream` 做法

在 FSDP1 中，`record_stream` 贯穿整个内存管理流程：

```python
# FSDP1 (_flat_param.py) 中的典型用法

# 1. All-gather 前：告诉 unshard stream "这个 sharded param 我还需要用"
_no_dispatch_record_stream(
    sharded_flat_param,
    self._device_handle.current_stream(),  # unshard_stream
)

# 2. 释放低精度分片：告诉当前 stream "这个 tensor 你还得用着"
_no_dispatch_record_stream(
    self.flat_param._mp_shard,
    self._device_handle.current_stream(),
)

# 3. 释放 unsharded param：防止提前回收
_no_dispatch_record_stream(
    unsharded_flat_param, 
    self._device_handle.current_stream()
)
```

### 3.2 `record_stream` 的工作原理

```text
record_stream 做的事：
┌─────────────────────────────────────────────────────────┐
│ PyTorch Caching Allocator                               │
│                                                         │
│  Block 0x7f00: [████████████]  ← Tensor A 在用         │
│  Block 0x7f10: [            ]  ← 空闲                  │
│                                                         │
│  正常情况： Tensor A 的 Python ref 被删除              │
│         → allocator 立即标记 Block 0x7f00 为"可复用"    │
│         → 下一个分配可能复用这块内存                    │
│                                                         │
│  record_stream(tensor, stream):                         │
│         → allocator 记录："Block 0x7f00 在 stream      │
│           上的所有操作完成前，不准复用！"               │
│         → 即使 Python ref 已删除，也要等 stream 完成    │
└─────────────────────────────────────────────────────────┘
```

**本质上，`record_stream` 是对 CUDA caching allocator 的一个"提示"**：告诉它某个 tensor 的内存虽然 Python 侧已经不持有了，但在特定 stream 上还有未完成的操作，不能回收。

### 3.3 `record_stream` 的三个根本问题

#### 问题 1：不确定性（Non-deterministic memory behavior）

```text
Python 的 GC 时机是不确定的 ──► record_stream 的调用时机也是不确定的

场景：
  Tensor A（all-gather 输出）在 Python 中被 del
  ├─ 情况 A：GC 立即触发 → record_stream 立即注册 → allocator 正确延迟回收
  └─ 情况 B：GC 延迟到下一轮迭代 → record_stream 晚注册
         → 在此期间 allocator 可能已经把内存给了别的 tensor
         → 新 tensor 写入 → 旧 stream 上的 kernel 读到脏数据 → 数值错误！

结果：同样的代码，跑 100 次可能 99 次正常、1 次出错
     这种 bug 极难复现和定位
```

#### 问题 2：`limit_all_gathers=True` 导致 CPU 阻塞

FSDP1 为了缓解 record_stream 的不确定性，引入了 `limit_all_gathers=True` 选项：

```python
# FSDP1 的 limit_all_gathers 逻辑（概念代码）
if self.limit_all_gathers:
    # 强制在 CPU 侧等待上一次 all-gather 完成
    # 确保 record_stream 在正确的时机生效
    self._all_gather_handle.wait()  # ← CPU 阻塞！
```

这虽然提高了正确性，但**破坏了计算与通信的重叠**——CPU 被阻塞等待 GPU 完成，违背了异步并行的初衷。

#### 问题 3：语义错位——内存生命周期 ≠ Stream 使用周期

```text
record_stream 解决的是："这个 tensor 的内存什么时候可以安全回收？"
而实际需要回答的是："这个 tensor 的数据什么时候不再被任何 stream 需要？"

这两个问题的答案不一定一致：

  tensor = all_gather_output
  compute_stream:  使用 tensor 做 fwd(m1) → 完成后不再需要
  ag_stream:       使用 tensor 写入 all-gather 结果 → 完成后不再需要

  record_stream(tensor, compute_stream) 告诉 allocator：
    "compute_stream 上的所有操作完成前不许回收"
  
  但实际上，只要 ag_stream 和 compute_stream 上的相关操作都完成了，
  就可以回收了。record_stream 的粒度太粗（整个 stream 的所有操作）。
```

### 3.4 FSDP2 的替代方案：手动同步 via CUDA Events

FSDP2 不依赖 allocator 的 stream 记录机制，而是**精确地控制同步点**：

```text
┌─────────────────────────────────────────────────────────┐
│              FSDP2 的手动同步方案                         │
│                                                         │
│  ┌──────────────────┐                                  │
│  │  引用计数 + Event │                                  │
│  │                  │                                  │
│  │  all_gather_state│──► 持有上一层 all-gather 结果     │
│  │  .event          │──► 标记 AG 操作完成的时间点       │
│  └────────┬─────────┘                                  │
│           │                                            │
│           ▼                                            │
│  下一层的 wait_for_unshard:                             │
│    1. 读 all_gather_state.event                       │
│    2. ag_stream.wait_event(event)  ← 精确同步         │
│    3. 清除 all_gather_state（释放引用）                │
│    4. 此时才可以安全回收上一层的输出内存               │
│                                                         │
│  关键：不依赖 GC 时机，不依赖 allocator hint            │
│       完全由代码逻辑显式控制                             │
└─────────────────────────────────────────────────────────┘
```

**六个关键同步点**（Event 的 record 和 wait）：

```text
                     时间轴 ──────────────────────────────────────►

Event 1: post_optim_event (optimizer → all-gather)
  │  optimizer step 完成后 record
  │  ├─ ag_copy_in_stream.wait_event(post_optim_event)
  │  └─ ag_stream.wait_event(post_optim_event)
  │  保证：参数更新完成后，才发起新的 all-gather

Event 2: all_gather_copy_out_event (all-gather → compute)
  │  all-gather copy-out 完成后 record
  │  └─ compute_stream.wait_event(all_gather_copy_out_event)
  │  保证：all-gather 数据就绪后，才开始前向计算

Event 3: AllGatherState.event (layer N → layer N+1)
  │  上一层的 all-gather copy-out 完成后 record
  │  └─ 下一层的 ag_copy_in_stream.wait_event(prev.event)
  │  保证：上一层的输出不再使用时，才释放其缓冲区

Event 4: _reshard_after_forward_event (post-forward reshard → next unshard)
  │  post-forward reshard 完成后 record
  │  └─ 下次 unshard 的 ag_stream.wait_event(this_event)
  │  保证：reshard 完成（数据已备份）后，才能覆盖 all-gather 缓冲区

Event 5: ReduceScatterState.event (RS → 后续)
  │  reduce-scatter 完成后 record
  │  └─ 后续 post_backward 的 compute_stream.wait_event(rs_event)
  │  保证：梯度 reduce-scatter 完成后，才能清理状态

Event 6: _post_reduce_event (梯度 offload → optimizer)
  │  梯度 D2H 拷贝完成后 record
  │  └─ CPU optimizer 线程等待此 event
  │  保证：梯度已到达 CPU，optimizer 才能 step
```

## 问题四：手动同步相比 `record_stream` 有什么具体好处？

### 4.1 确定性（Determinism）

```text
FSDP1 + record_stream:
  Python GC 时机 → record_stream 注册时机 → allocator 行为
  ├─ 受 GC 策略影响（引用计数 vs 分代 GC）
  ├─ 受 Python 运行时状态影响
  └─ 同样代码可能表现出不同的内存行为

FSDP2 + Event sync:
  代码显式控制 → 同步点精确 → 内存行为可预测
  ├─ 不依赖 GC
  ├─ 不依赖 allocator 内部状态
  └─ 同样的输入总是产生同样的内存使用模式
```

### 4.2 不需要 CPU 阻塞

```text
FSDP1 的 limit_all_gathers=True：
  CPU 线程必须 wait() → 阻塞 → 无法提交后续工作
  代价：牺牲了 CPU 的"提前量"（lead），减少 overlap

FSDP2 的 Event sync：
  CPU 只负责 record event 和下发 wait 指令
  GPU 自己按 event 的顺序执行
  CPU 可以一直跑到很远的前面去，最大化 overlap
```

**对比图示**：

```text
FSDP1 (limit_all_gathers=True):
  CPU: [submit AG1] [wait AG1 ░░░░] [submit AG2] [wait AG2 ░░░░]
  GPU: [AG1         ] [fwd1]        [AG2         ] [fwd2       ]
       └─ CPU 空等 ─┘                     └─ CPU 空等 ─┘

FSDP2 (Event sync):
  CPU: [submit AG1] [submit AG2] [submit AG3] [submit RS1] ...
  GPU: [AG1   ] [AG2   ] [AG3   ]
       [fwd1  ] [fwd2  ] [fwd3  ]
       └─ full overlap, no CPU blocking ─┘
```

### 4.3 更精确的同步粒度

```text
record_stream 的粒度：
  "stream 上所有操作完成前，不准释放"
  → 即使 stream 上后面的 kernel 根本不碰这块内存，也得等

Event sync 的粒度：
  "这个特定的 event 被 signal 后，就可以安全释放"
  → event 精确标记了"最后使用这块内存的操作"在哪里完成
  → 更早释放内存，降低峰值显存
```

### 4.4 不依赖 PyTorch Allocator 的内部行为

```text
record_stream 是把同步责任"外包"给 caching allocator
  ├─ allocator 的实现可能随 PyTorch 版本变化
  ├─ 不同后端的 allocator 行为可能不同 (CUDA vs MTIA vs HPU)
  └─ 调试困难：allocator 内部状态对用户不可见

Event sync 是 FSDP2 自己管理同步
  ├─ 同步逻辑在 FSDP 代码中显式可见
  ├─ 跨后端统一（只要后端支持 CUDA-style event/stream）
  └─ 调试友好：可以打印 event 状态，追踪同步链
```

## 问题五：能用一个具体的例子说明 FSDP2 内存管理器的完整工作流程吗？

以一个两层的简单模型为例：

```python
import torch
import torch.nn as nn
from torch.distributed.fsdp import fully_shard

class SimpleModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layer1 = nn.Linear(1024, 1024)
        self.layer2 = nn.Linear(1024, 1024)
    
    def forward(self, x):
        return self.layer2(self.layer1(x))

model = SimpleModel()
fully_shard(model.layer1)  # layer1 独立成组
fully_shard(model.layer2)  # layer2 独立成组
fully_shard(model)         # 根组（实际上为空，因为所有参数都已分配）
```

**前向传播时的内存管理器工作流程**：

```text
时间轴 ──────────────────────────────────────────────────────────►

Step 1: root_pre_forward
  │
  ├─ ag_copy_in_stream.wait_event(post_optim_event)
  │  └─ 确保 optimizer step 已完成，参数数据是最新的
  ├─ ag_stream.wait_event(post_optim_event)
  │  └─ 确保 AG stream 不会读到旧数据
  │
  ▼
Step 2: layer1.pre_forward → unshard()
  │
  ├─ 分配: layer1 的所有 all_gather_output
  │  │     storage.resize_(numel * world_size * element_size)
  │  │     此时: GPU 显存增加 ~(参数量 * world_size) 字节
  │  │
  ├─ Copy-In Stream:  将 layer1 的分片参数写入 output buffer
  ├─ AG Stream:       NCCL all-gather 填满 output
  ├─ AG Stream:       record all_gather_copy_out_event
  │
  ├─ wait_for_unshard():
  │  │  compute_stream.wait_event(all_gather_copy_out_event)
  │  │  确保 all-gather 完成后，compute stream 才能读
  │  │
  │  └─ layer1 的参数现在处于 UNSHARDED 状态
  │     本质上: 每个参数的 .data 指向 all_gather_output 的 as_strided view
  │
  ▼
Step 3: fwd(layer1) 在 Compute Stream 上执行
  │
  │  同时（并行）:
  │
  ├─ CPU 触发 layer2.pre_forward → unshard()
  │  │
  │  ├─ 检查 comm_ctx.all_gather_state:
  │  │  └─ 为 None（layer1 是第一个），直接继续
  │  │
  │  ├─ 分配: layer2 的 all_gather_output (显存再增加)
  │  ├─ AG Stream:     NCCL all-gather(layer2)
  │  ├─ AG Stream:     record event_layer2
  │  │
  │  └─ layer2 的 AG 与 layer1 的 fwd 重叠！
  │
  ▼
Step 4: layer1 的 all_gather_output 何时释放？
  │
  ├─ layer1.post_forward 中调用 reshard()
  │  └─ 但！如果 world_size > 1 且在 FORWARD 状态:
  │     不会立即释放！而是保存到 comm_ctx.all_gather_state
  │     comm_ctx.all_gather_state = AllGatherState(
  │         all_gather_result=layer1_result,
  │         event=layer1_copy_out_event
  │     )
  │
  ▼
Step 5: layer2.wait_for_unshard() — 释放 layer1 的输出
  │
  ├─ prev_state = comm_ctx.all_gather_state  # layer1 的
  ├─ ag_copy_in_stream.wait_event(prev_state.event)
  │  └─ 确保 layer1 的 copy-out 完成后，才允许 copy-in stream 前进
  ├─ comm_ctx.all_gather_state = None  # 清除引用
  │  └─ 如果 Python 侧没有其他引用 → allocator 可回收 layer1 的缓冲区
  │
  └─ 然后: layer2 的 copy-out 到 Compute Stream 的数据就绪
```

**内存峰值对比**：

```text
FSDP1 (无精心管理):
  显存 ┤          ╭──────╮
       │   ╭──────╯      ╰──────╮
       │   │ layer1 AG out     │ layer2 AG out
       │   │ + layer2 AG out   │ + ...
       └───┴───────────────────┴──────► 时间
           可能同时持有多个层的完整参数

FSDP2 (Event 精确同步):
  显存 ┤   ╭─╮      ╭─╮
       │ ╭─╯ ╰──────╯ ╰──────
       │ │ layer1 │ layer2 │
       └─┴────────┴────────┴──────► 时间
          每层用完立即（通过 event 同步后）释放
          峰值显存 ≈ 最大一层的完整参数
```

### 6.2 关键同步链（从源码提取）

```text
初始化时（lazy_init）:
  FSDPCommContext 创建专用 Stream:
    all_gather_copy_in_stream  ← 独立于 compute stream
    all_gather_stream          ← 独立于 compute stream
    reduce_scatter_stream      ← 独立于 compute stream
    all_reduce_stream          ← 独立于 compute stream (HSDP)

每轮迭代的同步链:

  optimizer.step()
    │
    ├─ 用户 record post_optim_event (可选)
    │
    ▼
  root_pre_forward:
    │
    ├─ if post_optim_event:
    │     ag_copy_in_stream.wait_event(post_optim_event)
    │     ag_stream.wait_event(post_optim_event)
    │  else:
    │     ag_copy_in_stream.wait_stream(current_stream)
    │     ag_stream.wait_stream(current_stream)
    │
    ▼
  param_group.unshard():
    │
    ├─ if _reshard_after_forward_event:
    │     ag_stream.wait_event(_reshard_after_forward_event)
    │     (确保上一轮的 post-forward reshard 已完成)
    │
    ├─ copy-in stream: foreach_copy(sharded → output buffer)
    ├─ ag_stream.wait_stream(copy_in_stream)
    ├─ ag_stream: NCCL all_gather
    ├─ ag_stream: record all_gather_copy_out_event
    │
    ▼
  param_group.wait_for_unshard():
    │
    ├─ if comm_ctx.all_gather_state (上一层的结果):
    │     copy_in_stream.wait_event(prev.event)
    │     comm_ctx.all_gather_state = None  ← 释放上一层
    │
    ├─ compute_stream.wait_event(all_gather_copy_out_event)
    │   (仅在 async_op=False 时)
    │
    ├─ 遍历 fsdp_params, 调用 init_unsharded_param()
    │   从 all_gather_output 创建 as_strided view
    │
    └─ 设置状态: SHARDED → UNSHARDED
    │
    ▼
  fwd(module) on compute stream ────────┐
    │                                    │ 并行！
    │  下一层的 pre_forward:             │
    │    unshard() → AG stream           │
    │                                    │
    ▼                                    ▼
  param_group.reshard() (post_forward):
    │
    ├─ if world_size > 1 and FORWARD:
    │     # 不立即释放！保存到 comm_ctx 给下一层
    │     comm_ctx.all_gather_state = AllGatherState(
    │         result=all_gather_result,
    │         event=all_gather_copy_out_event
    │     )
    │  else:
    │     # 立即等待并释放
    │     _wait_all_gather_streams_on_event(event)
    │
    └─ 参数回到 SHARDED 或 SHARDED_POST_FORWARD 状态
```

## 总结

FSDP2 的内存管理器通过以下设计实现了**确定性的、高性能的**多 Stream 内存管理：

| 机制 | 解决的问题 |
|------|-----------|
| **Storage Resizing** (`resize_(N)` / `resize_(0)`) | 不创建新 Tensor，保留 autograd 引用，避免 use-after-free |
| **专用 CUDA Stream**（AG / RS / Copy-In / All-Reduce） | 隔离不同类型操作，实现最大 overlap |
| **CUDA Event 精确同步**（6 个关键同步点） | 替代 `record_stream`，不依赖 GC 时机，确定性地控制内存生命周期 |
| **AllGatherState 跨层接力** | 上一层输出在被下一层"签收"后才释放，防止提前回收 |
| **引用计数 + Event 双重保护** | Python 侧引用防止 allocator 回收；Event 保证 Stream 间 happens-before 关系 |

**核心结论**：FSDP2 放弃 `record_stream` 的根本原因，是将内存管理的正确性从"依赖 PyTorch Allocator 的隐式行为 + GC 时机"转移到"FSDP 代码中的显式 Event 同步"。这带来了**确定性**、**不需要 CPU 阻塞**、**更精确的同步粒度**和**更好的跨后端可移植性**。

> 文档原文：*"FSDP2 implements a different memory management approach to handle the multi-stream usages that avoids torch.Tensor.record_stream. This ensures deterministic and expected memory usage and does not require blocking the CPU like in FSDP1's limit_all_gathers=True."*
>
> —— PyTorch 2.12 FSDP2 官方文档
